# GTD Todos UX Audit

Last updated: 2026-04-25 19:59 PDT

## 1) Concise diagnosis

### What is working
- Core GTD buckets are implemented (`inbox`, `next_action`, `project`, `waiting_for`, `calendar`, `someday_maybe`, `reference`).
- Clarify flow exists in `/todos/process` with keyboard shortcuts and one-by-one processing.
- Backend mapping is consistent (`processTodo` cleanly maps outcomes to bucket/item type changes).

### Why it feels wrong
- **Board-first mental model**: `/todos` is visually a multi-bucket board, so users think in folder management, not decision workflow.
- **Decision tree is implicit, not explicit**: `/todos/process` shows outcomes as flat buttons, but not the GTD question chain.
- **Capture → clarify separation is weak in UI hierarchy**: "Process Inbox" is present but not dominant as primary action.
- **Too many parallel surfaces** (tabs, sections, details panels) create organizational overhead instead of fast inbox clearing.
- **Project handling is list-centric** (project cards + linked actions), but not framed as "define successful outcome + next physical action".

Bottom line: implementation is structurally GTD-capable, but **interaction language is bucket taxonomy, not GTD reasoning**.

---

## 2) Proposed IA + interaction model

## North-star model: "Process-first GTD"

### Primary modes
1. **Process mode (default CTA)**
   - One inbox item at a time.
   - Guided questions in sequence.
2. **Engage mode**
   - Context-based execution view (primarily Next Actions + Calendar + Waiting snapshot).
3. **Review mode**
   - Weekly review dashboard across all buckets.

### GTD decision flow (UI sequence)
For each inbox item, show only the next decision:
1. Is it actionable?
   - No → Trash / Reference / Someday
   - Yes → continue
2. Is it <2 minutes?
   - Yes → Do now
   - No → continue
3. Is it delegated?
   - Yes → Waiting For (+ who/what)
   - No → continue
4. Is it time-specific?
   - Yes → Calendar (+ date/time)
   - No → continue
5. Is it multi-step?
   - Yes → Project (+ auto-create first next action)
   - No → Next Action

This should be visibly rendered as a guided flow, not a flat option grid.

## 3) Concrete UI recommendations

### A) `/todos` (dashboard) should become command center, not bucket board
- Keep summary counts, but reduce full bucket walls by default.
- Make **"Process Inbox" the hero CTA** if inbox > 0.
- Show only:
  - Inbox count + quick start
  - Today calendar items
  - Next actions by context
  - Waiting-for follow-ups due/oldest
- Move full bucket browsing under an "All buckets" expandable view.

### B) `/todos/process` should become a true decision wizard
- Replace flat outcome grid with **step-by-step decision cards**.
- Each step has 2–3 choices only, with short rationale text.
- Persist small metadata inline when needed (waiting note, calendar time, project next action).
- Show breadcrumb/progress like:
  - `Actionable? -> <2 min? -> Delegated? -> Time-specific? -> Outcome`

### C) Interaction and visual clarity
- One primary action per step (others secondary).
- Strong typographic hierarchy for current item content.
- Keep hotkeys, but surface them progressively (context-sensitive, not full legend overload).
- Add microcopy from GTD language:
  - "What’s the next physical action?"
  - "What outcome defines this project done?"

### D) Data/model improvements (minimal but high impact)
- On project conversion, prompt for (or auto-generate draft for) **first next action**.
- Add optional `review_at` / follow-up date for waiting-for (improves review behavior).
- Keep current bucket schema (no major migration needed for v1 UX fix).

## 4) Implementation roadmap

## Phase 1 — UX reframing (no schema migrations)
- Make `/todos` process-first (hero CTA + reduced board density).
- Rebuild `/todos/process` into explicit decision wizard.
- Keep existing process API and outcome mapping.

## Phase 2 — GTD quality-of-life
- Project conversion asks for first next action before finalizing.
- Waiting-for includes optional follow-up date.
- Better context filters in Engage view.

## Phase 3 — Review system
- Weekly review screen with checklist:
  - empty inbox
  - prune someday/maybe
  - stale waiting-for
  - projects missing next action
- Add review health score and stale-item indicators.

## Phase 4 — Polish + speed
- Improve keyboard flow for process wizard.
- Add "quick wins" lane (<2 min suggestions during processing).
- Tighten mobile-first interaction for one-thumb triage.

## 5) Example flows (3 tasks)

### Example A — "Reply to Sam about Tuesday meeting"
- Actionable? **Yes**
- <2 min? **Yes**
- Outcome: **Do now** (mark complete)
- Result: leaves inbox immediately, no list clutter.

### Example B — "Plan Q3 hiring pipeline"
- Actionable? **Yes**
- <2 min? **No**
- Delegated? **No**
- Time-specific? **No**
- Multi-step? **Yes**
- Outcome: **Project**
- Prompt: define first next action → "Draft role scorecard"
- Result: project created + linked next action.

### Example C — "Need legal to review contract"
- Actionable? **Yes**
- <2 min? **No**
- Delegated? **Yes**
- Outcome: **Waiting For**
- Add note: "Waiting on Priya legal review"
- Optional follow-up date
- Result: tracked in waiting list, visible in review.

---

## Recommended product direction
Adopt **process-first GTD** (wizard-like clarification) and demote bucket board to secondary navigation. This preserves current backend while fixing the core UX mismatch.
